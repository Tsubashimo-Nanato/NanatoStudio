## 6. ROS 2 Development Platform

### 6.1 The Underlay and Overlay Model

The official ROS 2 workspace tutorial explains that a workspace can act as an overlay on top of an underlay. In this project:

- your ROS 2 Foxy installation is the underlay
- `aiformula` is an overlay
- `pid_ws` is another overlay used for Sophia controller work

The official ROS 2 documentation also warns that building in a terminal that already has an overlay sourced can create hard-to-debug issues. That warning is highly relevant here.

### 6.2 Recommended Terminal Strategy

Use separate terminals for separate purposes:

- build terminal for `colcon build`
- runtime terminal for `source` and `ros2 launch`
- inspection terminal for `ros2 node`, `ros2 topic`, and `ros2 param`

This reduces many avoidable environment errors.

### 6.3 Minimal Build Workflow

#### Build the baseline workspace

```bash
source /opt/ros/foxy/setup.bash
cd <aiformula-workspace>
rosdep install -i --from-path src --rosdistro foxy -y
colcon build --symlink-install
```

#### Source the baseline workspace

```bash
source /opt/ros/foxy/setup.bash
cd <aiformula-workspace>
source install/local_setup.bash
```

#### Build and source the controller workspace

```bash
source /opt/ros/foxy/setup.bash
cd <pid-workspace>
colcon build --symlink-install
source install/local_setup.bash
```

### 6.4 How to Check What ROS 2 Sees

Use these commands first:

```bash
ros2 pkg list
ros2 node list
ros2 topic list
ros2 param list
```

If a package you expect is missing from `ros2 pkg list`, the problem is usually one of:

- the workspace was not built
- the workspace was not sourced
- the wrong workspace is active
- the package name is not what you think it is

### 6.5 How to Read a Running System

A good inspection sequence is:

```bash
ros2 node list
ros2 topic list -t
ros2 topic info /<topic_name>
ros2 node info /<node_name>
ros2 param list
```

For example:

```bash
ros2 node list
ros2 topic list -t
ros2 node info /motor_controller
ros2 topic echo /odom
```

You are not just looking for whether the system is alive. You are trying to answer:

- is the expected graph present?
- are the expected data types present?
- are publishers and subscribers connected?

### 6.6 Launch as the Default Way to Start Systems

The official ROS 2 launch tutorial correctly frames `ros2 launch` as the standard way to bring up a multi-node system. That is how this project should be used most of the time.

Typical pattern:

```bash
ros2 launch <package_name> <launch_name>.launch.py
```

Project examples include:

```bash
ros2 launch launchers all_nodes.launch.py
ros2 launch launchers our_all_nodes.launch.py
ros2 launch lane_line_publisher lane_line_publisher.launch.py
ros2 launch road_detector road_detector.launch.py
```

### 6.7 Package Execution

When a package exposes a specific executable, use `ros2 run`.

Examples from this project include:

```bash
ros2 run motor_controller motor_controller
ros2 run lane_points lane_0529oa
ros2 run kalman_filter withoutkalman_0312
ros2 run obsticle_avoidence b_spline
ros2 run trajectory_follower lya_oa
```

Use `ros2 run` when you want one focused node. Use `ros2 launch` when you want a composed system.

### 6.8 A Practical Baseline Bring-Up Flow

Here is a realistic baseline flow for a new team member:

```bash
# Terminal A
source /opt/ros/foxy/setup.bash
cd <aiformula-workspace>
source install/local_setup.bash

# Prepare hardware interfaces
./init_sensors.sh

# Start the baseline platform
ros2 launch launchers all_nodes.launch.py
```

Then in another terminal:

```bash
# Terminal B
source /opt/ros/foxy/setup.bash
cd <aiformula-workspace>
source install/local_setup.bash

ros2 node list
ros2 topic list -t
ros2 param list
```

And if you need Sophia controller logic:

```bash
# Terminal C
source /opt/ros/foxy/setup.bash
cd <pid-workspace>
source install/local_setup.bash

ros2 run trajectory_follower lya_follower_connected_omegat_global
```

### 6.9 TF and Visualization

The official TF documentation is especially relevant to this project because the baseline stack launches static transforms and sensor-related frames early.

Helpful tools:

```bash
ros2 run tf2_tools view_frames
rviz2
```

Use RViz not just for pretty visualization, but for quick answers to practical questions:

- is odometry moving?
- are frames consistent?
- is perception data appearing in the expected frame?

### 6.10 Common ROS 2 Mistakes in This Project

#### Building in the wrong workspace

You think a package is broken, but you actually built a different workspace.

#### Forgetting to source after rebuild

You rebuilt the code but are still running an old environment.

#### Starting a controller before the baseline graph exists

If the controller depends on topics that are not published yet, the issue is startup order, not necessarily controller logic.

#### Ignoring TF

Perception and control problems often come from frame issues rather than algorithmic logic.

### 6.11 A Good ROS 2 Debugging Habit

When something fails, answer these questions in order:

1. Is the package discoverable?
2. Is the node running?
3. Is the topic published?
4. Is the message type correct?
5. Are parameters loaded?
6. Is TF valid?
7. Is the dependency workspace sourced?

This discipline saves time.
